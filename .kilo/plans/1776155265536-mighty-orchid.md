# Frontend Design Improvements Plan

**Project:** Visions Transport Website v2
**Date:** 2026-04-14
**Status:** Planning Phase

---

## Executive Summary

This plan addresses critical UI/UX, accessibility, performance, and architectural improvements for the vision-website-v2 project. Changes are prioritized by impact and user experience value.

**Total Tasks:** 12
**Estimated Effort:** 3-4 days
**Priority Breakdown:** P0 (2), P1 (5), P2 (3), P3 (2)

## Priority Legend

- **P0 (Critical):** Blocks users from completing tasks, accessibility violations
- **P1 (High):** Significant UX friction, major performance issues
- **P2 (Medium):** Noticeable improvements, better maintainability
- **P3 (Low):** Nice-to-have, future-proofing

## Task Breakdown

### P0 - Critical Issues

#### Task 1: Add Floating Labels to VisionForm
**Category:** UX / Accessibility
**Files:** `src/components/pages/home/vision-form/index.tsx`, `src/app/globals.css`
**Estimated Time:** 2 hours

**Problem:** Form relies solely on placeholder text which disappears when typing, creating poor UX and accessibility issues.

**Implementation:**
1. Add `<label>` elements with `sr-only` class for screen readers
2. Implement floating label pattern using CSS:
   ```css
   .floating-label-group {
     @apply relative;
   }
   
   .floating-label {
     @apply absolute left-3 top-1/2 -translate-y-1/2 transition-all duration-200 pointer-events-none;
   }
   
   .floating-label-group input:focus ~ .floating-label,
   .floating-label-group input:not(:placeholder-shown) ~ .floating-label {
     @apply text-xs top-0 bg-white px-1;
   }
   ```
3. Update input fields to use placeholder only for visual pattern (e.g., " ")
4. Add `aria-describedby` linking inputs to error messages
5. Add `aria-invalid="true"` when validation fails

**Success Criteria:**
- [ ] Labels are visible at all times (floating or static)
- [ ] Screen readers announce both label and error messages
- [ ] Form passes WCAG 2.1 AA standards

---

#### Task 2: Add ARIA Attributes to FAQ Accordion
**Category:** Accessibility
**Files:** `src/components/shared/Faq/index.tsx`
**Estimated Time:** 1 hour

**Problem:** FAQ buttons lack proper ARIA attributes for screen reader compatibility.

**Implementation:**
1. Add `aria-expanded={item.open}` to each button
2. Add `aria-controls={`faq-panel-${index}`}` to buttons
3. Add `id={`faq-panel-${index}`}` to answer divs
4. Add `role="region"` to the FAQ container
5. Add `aria-labelledby` to answer panels

**Code Changes:**
```tsx
<button
  type="button"
  aria-expanded={item.open}
  aria-controls={`faq-panel-${index}`}
  className="..."
>
  {/* ... */}
</button>

<div
  id={`faq-panel-${index}`}
  role="region"
  aria-labelledby={`faq-button-${index}`}
  className={...}
>
  {/* answer content */}
</div>
```

**Success Criteria:**
- [ ] Screen readers correctly announce expanded/collapsed state
- [ ] Keyboard navigation works smoothly
- [ ] Passes axe-core accessibility audit

### P1 - High Priority

#### Task 3: Smooth FAQ Accordion Animations
**Category:** UX / Animation
**Files:** `src/components/shared/Faq/index.tsx`
**Estimated Time:** 1.5 hours

**Problem:** Accordion uses `block/hidden` causing jarring layout shifts.

**Implementation:**
1. Replace conditional rendering with CSS Grid animation:
   ```tsx
   <div
     className="grid transition-all duration-300 ease-in-out"
     style={{
       gridTemplateRows: item.open ? '1fr' : '0fr'
     }}
   >
     <div className="overflow-hidden">
       {/* answer content */}
     </div>
   </div>
   ```
2. Or use Framer Motion:
   ```tsx
   <motion.div
     initial={false}
     animate={{ height: item.open ? 'auto' : 0 }}
     transition={{ duration: 0.3, ease: 'easeInOut' }}
     className="overflow-hidden"
   >
     {/* answer content */}
   </motion.div>
   ```

**Success Criteria:**
- [ ] Smooth height transition on open/close
- [ ] No layout jumps
- [ ] Performance remains good (60fps)

#### Task 7: Semantic Typography Refactoring
**Category:** Accessibility / Maintainability
**Files:** `src/components/ui/Typography.tsx`
**Estimated Time:** 3 hours

**Problem:** Variant naming couples visual weight with HTML tags, confusing for screen readers.

**Implementation:**
1. Refactor variant naming to separate concerns:
   ```typescript
   type TypographyVariant =
     | "display" | "heading-1" | "heading-2" | "heading-3"
     | "heading-4" | "body-large" | "body" | "caption";
   
   type TypographyWeight = "regular" | "medium" | "semibold" | "bold";
   
   interface TypographyProps {
     variant: TypographyVariant;
     weight?: TypographyWeight;
     as?: "h1" | "h2" | "h3" | "h4" | "h5" | "h6" | "p" | "span" | "small";
     // ... existing props
   }
   ```

2. Create new mapping functions:
   ```typescript
   const getStyles = (variant: TypographyVariant, weight: TypographyWeight) => {
     // Map variant + weight to Tailwind classes
   };
   
   const getDefaultTag = (variant: TypographyVariant): HTMLHeadingTag => {
     switch(variant) {
       case 'display': return 'h1';
       case 'heading-1': return 'h1';
       case 'heading-2': return 'h2';
       // ...
       default: return 'p';
     }
   };
   ```

3. Update usage throughout codebase (gradual migration)
4. Deprecate old variants with warning logs

**Migration Strategy:**
- Phase 1: Add new system alongside old (no breaking changes)
- Phase 2: Update high-traffic pages (home, services, contact)
- Phase 3: Remove old variants

**Success Criteria:**
- [ ] New Typography works with `as` prop
- [ ] Backward compatible with existing usage
- [ ] Improved screen reader experience

### P2 - Medium Priority

#### Task 8: Expand Color Theme Variables
**Category:** Architecture / Theming
**Files:** `src/app/globals.css`
**Estimated Time:** 2 hours

**Problem:** Rigid hex colors make hover states and theming difficult.

**Implementation:**
1. Define color scales using HSL:
   ```css
   :root {
     --primary-h: 359;
     --primary-s: 87%;
     --primary-l: 52%;
     
     --primary-50: hsl(var(--primary-h), var(--primary-s), 95%);
     --primary-100: hsl(var(--primary-h), var(--primary-s), 90%);
     --primary-200: hsl(var(--primary-h), var(--primary-s), 80%);
     --primary-300: hsl(var(--primary-h), var(--primary-s), 70%);
     --primary-400: hsl(var(--primary-h), var(--primary-s), 60%);
     --primary-500: hsl(var(--primary-h), var(--primary-s), 52%); /* base */
     --primary-600: hsl(var(--primary-h), var(--primary-s), 45%);
     --primary-700: hsl(var(--primary-h), var(--primary-s), 38%);
     --primary-800: hsl(var(--primary-h), var(--primary-s), 31%);
     --primary-900: hsl(var(--primary-h), var(--primary-s), 24%);
     
     /* Repeat for secondary, gray, etc. */
   }
   ```

2. Update Tailwind config to use CSS variables
3. Update existing components to use new scales

**Success Criteria:**
- [ ] Full color scale available (50-900)
- [ ] Hover states use darker/lighter variants naturally
- [ ] Easier to implement dark mode in future

### P3 - Low Priority

#### Task 11: Add Reduced Motion Support
**Category:** Accessibility / UX
**Files:** `src/app/globals.css`, `src/components/shared/FadeIn.tsx`
**Estimated Time:** 1 hour

**Problem:** No global reduced motion support.

**Implementation:**
1. Add CSS media query:
   ```css
   @media (prefers-reduced-motion: reduce) {
     *,
     *::before,
     *::after {
       animation-duration: 0.01ms !important;
       animation-iteration-count: 1 !important;
       transition-duration: 0.01ms !important;
     }
   }
   ```

2. Update FadeIn component:
   ```tsx
   const prefersReducedMotion = useReducedMotion();
   
   if (prefersReducedMotion) {
     return <div {...props}>{children}</div>;
   }
   ```

**Success Criteria:**
- [ ] Respects user's motion preferences
- [ ] All animations disabled when preferred
- [ ] Content still accessible

#### Task 12: Add Test Mode for Animations
**Category:** Testing / CI
**Files:** `src/components/shared/FadeIn.tsx`
**Estimated Time:** 30 minutes

**Problem:** Automated screenshot tests fail with whileInView animations.

**Implementation:**
1. Add query param support:
   ```tsx
   import { useSearchParams } from 'next/navigation';
   
   const searchParams = useSearchParams();
   const disableAnimations = 
     searchParams.get('no-animations') === 'true' ||
     process.env.NODE_ENV === 'test';
   
   if (disableAnimations) {
     return <div {...props}>{children}</div>;
   }
   ```

2. Update CI test scripts to use `?no-animations=true`

**Success Criteria:**
- [ ] Screenshot tests pass consistently
- [ ] No regression in production animations
- [ ] Easy to toggle for debugging
